[#prod-glossario]
=== Glossário

[cols="1,3",options="header"]
|===
| Termo | Significado

| Worker
| Unidade executável servida pelo runtime. Vive em um diretório próprio dentro
  de `RUNTIME_WORKER_DIRS` e pode ter manifest, entrypoint, versão e kind.

| Worker manifest
| Arquivo `manifest.yaml` / `manifest.yml` que descreve `name`, `version`,
  `entrypoint`, `kind`, TTL, timeouts, public routes, cron e outros metadados.
  Quando ausente, o loader tenta `package.json` ou entrypoints convencionais.

| Execution kind
| Modo de execução normalizado: `FetchHandler`, `RoutesTable`, `StaticSpa`,
  `WasmModule` ou `Fullstack`. O kind pode vir do manifest ou ser inferido pelo
  entrypoint.

| Orchestrator
| Camada Rust em `edger-orchestrator` que compõe HTTP server, auth, router,
  registry de extensões, pipeline e `WorkerPool`.

| Isolate
| Trait de execução definida em `edger-core`. Implementações atuais incluem
  `DenoIsolate`, `WasmIsolate` e `MockIsolate` para testes.

| Deno CLI bridge
| Backend JS/TS funcional v1. Executa `deno run --no-check --no-prompt` por
  request com sandbox explícito (`--allow-read=<worker_dir>`, net configurável
  via `EDGER_DENO_ALLOW_NET`, write/run/ffi/sys negados), dentro do diretório
  do worker, usando `deno.json` / `deno.jsonc` local quando presente.

| Processo Deno persistente
| Backend JS/TS padrão. Mantém um processo Deno por worker, importa o módulo
  uma vez e despacha por socket Unix. A rede usa `manifest.allowNet` ou
  `EDGER_DENO_ALLOW_NET`; sem allowlist explícita, segue aberta por
  compatibilidade. O cache Deno usa `denoCacheMode: per-worker` por padrão para
  evitar compartilhamento de `DENO_DIR` entre workers, com `shared` disponível
  como trade-off operacional.

| deno_core
| Alternativa embutida avaliada para JS/TS. O caminho de produção atual é o
  processo Deno persistente por worker; `deno_core` só voltaria como backend
  interno futuro se densidade/latência justificarem o custo.

| Wasm ABI v2
| Contrato para módulos Wasm: exports `memory`, `edger_alloc(len)` e
  `edger_handle(ptr, len)`. Request e response trafegam por frames em linear
  memory.

| WorkerPool
| Componente de `edger-worker` que cria, cacheia, supervisiona e despacha
  requests para isolates.

| Extension
| Trait base para extensões Rust. Crates `edger-ext-*` devem escolher um modo:
  `Middleware`, `AuthProvider` ou `WorkerHandler`.

| AuthProvider
| Extensão responsável por resolver API keys em `ApiKeyPrincipal`. A extensão
  atual é `edger-ext-auth`.

| Middleware
| Extensão com hooks `on_request` / `on_response`. `edger-ext-gateway` é o
  template inicial de middleware pass-through.

| ApiKeyPrincipal
| Principal autenticado com `role`, `permissions`, `namespaces`, prefixo de key
  e flag `is_root`.

| Namespace
| Escopo de acesso para workers namespaced, como `@acme/app`. O root principal
  acessa todos os namespaces.

| publicRoutes
| Configuração compatível com Buntime para bypass de auth e hooks em rotas
  públicas específicas.

| Shell
| Camada planejada para micro-frontends e SPAs, incluindo injeção de
  `<base href>`. O protocolo dedicado ainda está em skeleton.

| Cron
| Scheduler nativo planejado para disparar requests internas declaradas em
  `manifest.cron[]`. Ainda não está implementado.
|===
