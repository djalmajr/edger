=== Modelo de domínio e manifests

O modelo público do runtime é centrado em workers, manifests, configs
normalizadas e kinds de execução. `edger-core` é a fonte de verdade desses
tipos.

==== WorkerManifest

`WorkerManifest` representa o arquivo humano `manifest.yaml` /
`manifest.yml`. Quando não há manifest, o loader tenta `package.json` e depois
entrypoints convencionais na ordem `index.html`, `index.ts`, `index.js`,
`index.mjs`, `index.wasm`, `index.wat`.

[cols="1,3",options="header"]
|===
| Campo | Significado

| `name`
| Nome do worker. Pode ser simples (`hello-world`) ou namespaced
  (`@team/checkout`).

| `version`
| Versão semver ou `latest` por default.

| `enabled`
| `false` remove o worker do index.

| `entrypoint`
| Arquivo executável, como `index.html`, `index.ts`, `index.js`, `index.wasm`
  ou `index.wat`.

| `kind`
| Hint explícito: `serverless`, `fetch`, `routes`, `backend`, `spa`, `static`,
  `wasm`, `ssr` ou `fullstack`.

| `ttl`, `timeout`, `idleTimeout`, `maxRequests`
| Configurações de lifecycle e timeout normalizadas para milissegundos.

| `maxBodySize`
| Limite de body parseado de valores como `10mb`.

| `publicRoutes`
| Bypass de auth/hooks para rotas públicas do worker.

| `cron`
| Jobs planejados para o scheduler nativo. Cada item usa `schedule`, `path` e
  `method` opcional.

| `base`
| Base de plugin/micro-frontend, usada para precedência de roteamento.

| `env`, `envPrefix`
| Ambiente do worker. O uso seguro ainda depende do backend de execução e do
  hardening da Fase 7.

| `injectBase`
| Habilita injeção de `<base href>` para SPA/shell. StaticSpa production ainda
  está pendente.
|===

==== WorkerConfig normalizado

`parse_worker_config` converte manifest em `WorkerConfig` com defaults:

[cols="1,2",options="header"]
|===
| Campo | Default / regra

| `enabled`
| `true`.

| `ttl_ms`
| `0`; worker efêmero quando TTL é zero.

| `timeout_ms`
| `30000`.

| `idle_timeout_ms`
| `60000`.

| `max_requests`
| `0`.

| `inject_base`
| `true`.

| `visibility`
| `protected`.

| `kind`
| Inferido por `kind`, extensão do entrypoint ou `FetchHandler`.
|===

==== Cron

O scheduler nativo lê `cron[]` no startup e dispara requests internas para o
worker resolvido. O `path` é relativo ao worker, mas deve começar com `/`.
`method` usa `GET` por default.

[source,yaml]
----
name: cron-worker
version: "1.0.0"
entrypoint: index.ts
kind: fetch
cron:
  - schedule: "*/1 * * * *"
    path: /tick
    method: POST
----

No v1, `schedule` aceita `@every <duration>` para intervalos curtos locais e
testes (`@every 25ms`, `@every 5s`) e cron simples de minuto (`* * * * *`,
`*/N * * * *` ou minuto fixo). Schedules inválidos falham no startup com erro
`CRON_SCHEDULE_INVALID`.

Requests cron entram no pipeline com `x-edger-internal: true`,
`Authorization: Bearer $ROOT_API_KEY` e `x-request-id: cron-...`. A credencial
root é removida antes de serializar a request para o worker; user code recebe o
marcador interno, mas não recebe o bearer root. Se houver jobs cron e
`ROOT_API_KEY` não estiver configurado, o servidor falha no startup com
`CRON_AUTH_MISSING`. Workers `enabled: false` não registram jobs; habilitar um
worker em runtime exige reload/restart para entrar no scheduler.

==== ExecutionKind

[cols="1,3",options="header"]
|===
| Kind | Uso

| `FetchHandler`
| Caminho JS/TS principal. Suporta `Deno.serve` e default export fetch pela Deno
  CLI bridge.

| `RoutesTable`
| Planejado para tabela de rotas/backend. Ainda não é production específico.

| `StaticSpa`
| SPA estática com leitura segura e injeção opcional de `<base href>`.

| `WasmModule`
| Executa `.wasm` ou `.wat` via `wasmtime`.

| `Fullstack`
| Reservado para adapters SSR/fullstack. Ainda não implementado.
|===

==== Endereçamento e resolução

O router aceita:

[source]
----
/name
/name@1.2.3
/@scope/name
/@scope/name@1.2.3
----

Regras relevantes:

* `/health`, `/ready`, `/api/*` e `/.well-known/*` são paths reservados.
* `latest` escolhe a maior versão semver disponível; se só houver `latest`, usa
  `latest`.
* Namespaces vêm do formato `@scope/name`.
* A request entregue ao worker recebe path reescrito sem o prefixo do worker.
* O header `x-base` preserva o base público, como `/hello-world`.

==== Invariantes

* `edger-core` não lê filesystem nem rede.
* Um worker precisa de `name` não vazio depois da resolução do manifest.
* Duplicidade de `(name, version)` no `ManifestIndex` gera `COLLISION`.
* Crates `edger-ext-*` dependem somente de `edger-core`.
* O runtime ativo não pode depender do adapter Bun removido.
