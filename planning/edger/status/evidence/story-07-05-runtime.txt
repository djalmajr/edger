## Behavior covered

- `WasmIsolate::execute_wasm` loads `.wasm`/`.wat` from the worker directory and
  rejects entrypoints that escape the worker root.
- Wasm module bytes must have Wasm magic bytes and stay under the configured
  module size cap.
- ABI v1 reads response body bytes from exported `memory` offset `0` using
  `http_status()` and `http_body_len()`.
- Host imports are rejected with `WASM_IMPORT_DENIED`.
- WASI imports are rejected with `WASI_IMPORT_DENIED` under deny-all sandbox
  mode.
- `WasiConfig::from_worker_config` filters sensitive env keys before future env
  injection.
- `WorkerPool::fetch_worker` dispatches Wasm through `WorkerConfig.kind` even
  without a caller-provided kind hint.
- The orchestrator runtime factory selects `WasmIsolate` for `WasmModule`.
- A single process/pool can serve a Deno JS worker and a Wasm worker without
  shared mutable backend state.
- `workers/wasm-hello/index.wat` is the source fixture and documents optional
  materialization of `index.wasm`.
- Host WASI preopens and request/response linear-memory ABI remain explicit
  follow-ups.
